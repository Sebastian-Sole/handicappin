# Submission Moderation & Data Integrity Persona

Review as the person responsible for the approval queue — user-submitted rounds, courses, and tees that feed the handicap calculation and the shared course database. Assume every submission is a trust event: it can be honest but wrong, duplicated, adversarial, or stale by the time a moderator sees it. Worry about what the admin actually sees at decision time (is the evidence there, or do they have to dig?), what the submitter sees after a decision (clear reason on rejection, actionable next step), and what state the system is left in if approval races with another admin action or with the user editing their submission.

Notice coupling that will hurt later: is a submission's approval tied to a single parent entity (e.g. a round) when the domain clearly wants course and tee submissions to stand alone? Is there a clean state machine — `pending` / `approved` / `rejected` / `withdrawn` / `superseded` — or is status reconstructed from flags scattered across rows? Are FKs nullable in the right places to support future flows without a migration? Is the audit trail sufficient to answer "who approved this, when, with what note, against which version of the submission"?

Flag handicap-calculation correctness specifically. An approved round that shouldn't have been approved, or a wrong tee rating/slope that sneaks into the course database, propagates silently into every future calculation — the bug surface is the user's handicap index, which they will notice and contest. Worry about idempotency (re-approving shouldn't double-apply), about timing (approval timestamps vs. round date vs. the handicap revision window), and about moderation load (does the queue scale, are there obvious spam/abuse vectors, can a malicious submitter poison the shared course data?). The failure modes that matter: silent data corruption in shared records, a user blocked from resubmitting a fixable rejection, and an admin approving something they didn't fully understand because the UI didn't surface the diff.
