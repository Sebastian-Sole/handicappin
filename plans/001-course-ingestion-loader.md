## Why this matters

An active ingestion effort is scraping Scottish golf courses (539 targets; Norway next) into per-course SQL files, but the pipeline dead-ends before production: the staged SQL inserts courses with `approvalStatus = 'pending'`, and the only user-facing course query filters to `approved` — so importing all 539 courses today would produce 539 rows **invisible to every user**. There is also no committed script that applies the staged SQL at all. This plan makes the ingestion pipeline actually deliver: a committed, idempotent loader and an explicit provenance/trust model.

## Steps

### Step 1: Add the provenance column to `course`

In `apps/web/db/schema.ts`, add to the `course` table (after `submittedBy`, `:118`):

```ts
source: text().$type<"user" | "ingest">().default("user").notNull(),
```

(No enum — repo convention is union-typed text; `submissions.submissionType` at `schema.ts:387` is the exemplar.)

Create `supabase/migrations/<timestamp>_course_provenance.sql` (timestamp = current UTC `YYYYMMDDHHMMSS`, strictly greater than the latest existing migration filename):

```sql

  add column if not exists "source" text not null default 'user';
```

### Step 3: Write the idempotent loader `scripts/load-staged-courses.mjs`

Requirements:

- Reads every `scripts/sql/_staging/*.sql`, in filename order.
- **Idempotency**: before executing a file, parse the course identity from its first `insert into public.course` statement (name, city, country are literal strings in the staged format — see the Brodick excerpt above) and skip the file (log `SKIP <file>: already loaded`) if a row already exists matching the `(name, country, city)` unique key. This makes re-runs safe without editing the staged SQL.
- **Provenance**: after executing a file, `update public.course set "source" = 'ingest' where ...` the same `(name, country, city)` key. (Do not edit the staged files to add the column — they are generator output; the loader owns provenance.)
- `--dry-run` flag (default OFF): print what would be loaded/skipped, execute nothing. Must degrade gracefully with no reachable database: when the connection env var is unset or the connection fails, still parse and list every file with its extracted `(name, country, city)` identity, mark existence as `unknown (no DB)`, and exit 0 — this makes identity-parsing verifiable in environments without a local stack.
- `--approve` flag (default OFF): after loading a file, set `"approvalStatus" = 'approved'` on the loaded course and its tees (`public."teeInfo"` rows whose `"courseId"` matches). **Default is to load as `pending`** — the operator reviews a batch (spot-check ratings/pars against the club site) and re-runs with `--approve`, or approves via the admin console (plans/002). This is the human quality gate; do not remove it.
- Connection (decided 2026-07-03 after first execution attempt): the repo's REST-based script convention (`reconcile-billing.mjs`, `seed-native-test-rounds.mjs`) cannot execute anonymous `do $$` blocks — PostgREST doesn't run arbitrary SQL and no `exec_sql` RPC exists. Add `"postgres": "^3.4.8"` (the exact version spec `apps/web/package.json:68` uses) to the ROOT `package.json` as a devDependency, and connect via the same env variable `apps/web/db/index.ts` uses (read that file for the exact name — likely `DATABASE_URL`). Run `pnpm install` with the pinned pnpm (`pnpm --version` must be 10.33.0 before installing) and verify the lockfile diff is additive-only with `overrides`/`patchedDependencies` preserved.
- Wrap each file in a transaction; on error, roll back that file, log `FAIL <file>: <error>`, continue with the next, and exit non-zero at the end if any file failed.
- Add a `load:courses` script to root `package.json`: `"load:courses": "node scripts/load-staged-courses.mjs"`.

**Verify** (staged file count is 38 as of 2026-07-03 — ingestion kept running after planning):
1. Without a DB: `node scripts/load-staged-courses.mjs --dry-run` → lists all 38 files with parsed identities, exit 0.
2. With a local Supabase stack (if available): full run → 38 loaded; re-run → 38 skipped (idempotency); `select count(*) from course where source = 'ingest'` → 38. If no stack is available, deliver with the no-DB dry-run verified and report live verification as pending — this is an accepted gap, not a STOP.
